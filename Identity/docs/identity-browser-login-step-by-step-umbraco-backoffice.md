# Umbraco Backoffice Login Step By Step: `localhost:5321/umbraco` -> `app.duchihao.com/identity`

This document describes the case where:

- browser entry URL is `https://localhost:5321/umbraco`
- client application is Umbraco CMS backoffice
- custom login page and issuer are:
  - `https://app.duchihao.com/identity`

This flow is different from the Identity SPA flow. Umbraco starts its own backoffice external-login flow first, then ASP.NET Core OpenID Connect middleware redirects the browser to the Identity issuer.

## Main Files

Identity workspace:

- `D:\OpenSaur\Identity\src\OpenSaur.Identity.Web\Features\Auth\Oidc\OidcEndpoints.cs`
- `D:\OpenSaur\Identity\src\OpenSaur.Identity.Web\Features\Auth\AuthSessionPrincipalFactory.cs`
- `D:\OpenSaur\Identity\src\OpenSaur.Identity.Web\Features\Auth\Oidc\IssuerAuthenticationFlow.cs`

Umbraco workspace:

- `D:\OpenSaur\Umbraco\src\OpenSaur.Umbraco.Web\Program.cs`
- `D:\OpenSaur\Umbraco\src\OpenSaur.Umbraco.Web\Authentication\OpenSaurBackOfficeAuthenticationExtensions.cs`
- `D:\OpenSaur\Umbraco\src\OpenSaur.Umbraco.Web\Authentication\OpenSaurBackOfficeUserProvisioningService.cs`
- `D:\OpenSaur\Umbraco\src\OpenSaur.Umbraco.Web\Authentication\OpenSaurIdentitySession.cs`

## Step 1: Browser opens Umbraco backoffice

URL:

- `GET https://localhost:5321/umbraco`

Umbraco entry:

- `D:\OpenSaur\Umbraco\src\OpenSaur.Umbraco.Web\Program.cs`

What happens:

1. Umbraco boots the backoffice middleware with `UseBackOffice()`.
2. The browser loads the Umbraco backoffice shell.
3. Umbraco checks whether a valid backoffice auth session already exists.

## Step 2: Umbraco starts its external login flow

Browser request:

- `GET https://localhost:5321/umbraco/management/api/v1/security/back-office/authorize?...identity_provider=Umbraco.OpenSaur.Identity`

Registration code:

- `D:\OpenSaur\Umbraco\src\OpenSaur.Umbraco.Web\Authentication\OpenSaurBackOfficeAuthenticationExtensions.cs`

What happens:

1. Umbraco selects the external provider:
   - `Umbraco.OpenSaur.Identity`
2. That provider was registered with `AddOpenIdConnect(...)`.
3. ASP.NET Core OpenID Connect middleware starts a challenge for that provider.

Important configuration values:

- `Authority = https://app.duchihao.com/identity`
- `ClientId = umbraco-web`
- `CallbackPath = /signin-oidc`
- scopes
- PKCE settings

## Step 3: OpenID Connect middleware redirects browser to the Identity issuer

Browser redirect:

- `GET https://app.duchihao.com/identity/connect/authorize?client_id=umbraco-web&redirect_uri=https%3A%2F%2Flocalhost%3A5321%2Fsignin-oidc&response_type=code&...`

Why this happens:

1. The OIDC middleware reads discovery metadata from:
   - `https://app.duchihao.com/identity/.well-known/openid-configuration`
2. It gets the authorization endpoint:
   - `https://app.duchihao.com/identity/connect/authorize`
3. It builds the full authorize URL using the Umbraco OIDC settings.
4. It returns an HTTP `302` redirect.
5. The browser follows that redirect.

Important point:

- this redirect is generated by ASP.NET Core OpenID Connect middleware
- there is no manual `window.location` call in your Umbraco code for this step

## Step 4: Identity issuer receives the authorize request

Browser request:

- `GET https://app.duchihao.com/identity/connect/authorize?client_id=umbraco-web&redirect_uri=https%3A%2F%2Flocalhost%3A5321%2Fsignin-oidc&response_type=code&...`

Identity backend code:

- `D:\OpenSaur\Identity\src\OpenSaur.Identity.Web\Features\Auth\Oidc\OidcEndpoints.cs`

What happens:

1. Identity resolves the OpenID Connect request from the HTTP context.
2. Identity checks whether the current browser already has a valid local Identity auth cookie.
3. If the user is not logged in on `app.duchihao.com`, Identity cannot issue the code yet.

## Step 5: Identity challenges the local login page

Identity backend code:

- `D:\OpenSaur\Identity\src\OpenSaur.Identity.Web\Features\Auth\Oidc\IssuerAuthenticationFlow.cs`

Method:

- `ChallengeIssuerLogin(HttpContext httpContext)`

What happens:

1. Identity takes the current request path and query string:
   - `https://app.duchihao.com/identity/connect/authorize?...`
2. It stores that authorize URL as the post-login return target.
3. It challenges the local Identity application auth scheme.
4. The browser is redirected to the custom login page on the same host:
   - `https://app.duchihao.com/identity/login?returnUrl=...`

Important point:

- Identity does not redirect to Umbraco at this stage
- it redirects only to its own local login page first

## Step 6: User signs in on the custom Identity login page

Identity frontend/backend code:

- `D:\OpenSaur\Identity\src\OpenSaur.Identity.Web\client\src\pages\login\LoginPage.tsx`
- `D:\OpenSaur\Identity\src\OpenSaur.Identity.Web\Features\Auth\AuthEndpoints.cs`

Browser request for login API:

- `POST https://app.duchihao.com/identity/api/auth/login`

What happens:

1. User enters credentials on the OpenSaur Identity login page.
2. The login API validates credentials.
3. Identity issues the local app auth cookie on `app.duchihao.com`.
4. The login page now has a valid local Identity session.

## Step 7: Identity login page resumes the original authorize request

Identity frontend code:

- `D:\OpenSaur\Identity\src\OpenSaur.Identity.Web\client\src\pages\login\LoginPage.tsx`
- `D:\OpenSaur\Identity\src\OpenSaur.Identity.Web\client\src\features\auth\utils\firstPartyOidc.ts`

Important helper:

- `isIssuerAuthenticationContinuationReturnUrl(...)`

What happens:

1. The login page reads its `returnUrl`.
2. It detects that the `returnUrl` is an authorization-continuation route, not a normal shell page.
3. The page calls:
   - `window.location.assign(...)`
4. The browser goes back to:
   - `https://app.duchihao.com/identity/connect/authorize?...`

Important point:

- the login page still does not redirect directly to Umbraco
- it only resumes the issuer-side authorize request

## Step 8: Identity authorize endpoint now issues the authorization code

Identity backend code:

- `D:\OpenSaur\Identity\src\OpenSaur.Identity.Web\Features\Auth\Oidc\OidcEndpoints.cs`
- `D:\OpenSaur\Identity\src\OpenSaur.Identity.Web\Features\Auth\AuthSessionPrincipalFactory.cs`

What happens:

1. The browser is now back on:
   - `https://app.duchihao.com/identity/connect/authorize?...redirect_uri=https://localhost:5321/signin-oidc...`
2. This time, Identity sees a valid local auth cookie.
3. Identity loads:
   - user
   - workspace
   - roles
   - permissions
4. Identity builds the OIDC `ClaimsPrincipal`.
5. Identity returns:
   - `Results.SignIn(principal, OpenIddictServerAspNetCoreDefaults.AuthenticationScheme)`

What OpenIddict does next:

1. It reads the original OIDC request parameters.
2. That request already contains:
   - `redirect_uri=https://localhost:5321/signin-oidc`
   - `state=...`
3. OpenIddict generates the authorization code.
4. OpenIddict returns an HTTP redirect to the original callback URI.

Browser redirect:

- `GET https://localhost:5321/signin-oidc?code=...&state=...`

This is the step that moves the browser from the Identity host back to the Umbraco host.

## Step 9: Umbraco receives the callback at `/signin-oidc`

Browser request:

- `GET https://localhost:5321/signin-oidc?code=...&state=...`

Umbraco OIDC middleware code path:

- registered in `OpenSaurBackOfficeAuthenticationExtensions.cs`

What happens:

1. ASP.NET Core OpenID Connect middleware processes the callback.
2. It exchanges the authorization code for tokens.
3. It invokes the configured `OnTokenValidated` event.

## Step 10: Umbraco copies claims and validates access

Umbraco code:

- `D:\OpenSaur\Umbraco\src\OpenSaur.Umbraco.Web\Authentication\OpenSaurBackOfficeAuthenticationExtensions.cs`
- `D:\OpenSaur\Umbraco\src\OpenSaur.Umbraco.Web\Authentication\OpenSaurIdentitySession.cs`

What happens:

1. Umbraco reads the access token returned by Identity.
2. It parses the JWT locally.
3. It copies important claims into the backoffice sign-in principal:
   - role
   - permissions
   - workspace id
   - impersonation claims
4. It normalizes standard login claims:
   - `NameIdentifier`
   - `Name`
   - `Email`
5. It validates that the user has one of:
   - `SUPER ADMINISTRATOR`
   - `Umbraco.CanManage`

If those claims are missing, the backoffice login is rejected here.

## Step 11: Umbraco provisions or synchronizes the backoffice user

Umbraco code:

- `D:\OpenSaur\Umbraco\src\OpenSaur.Umbraco.Web\Authentication\OpenSaurBackOfficeUserProvisioningService.cs`

Methods involved:

- `EnsureWorkspaceGroup(...)`
- `PrepareAutoLinkedUser(...)`
- `SynchronizeUser(...)`

What happens:

1. Umbraco ensures that a user group exists for the OpenSaur workspace.
2. If this is the first successful login, Umbraco auto-links or creates the local backoffice user.
3. It assigns the user to:
   - the workspace-derived group
   - `Administrators` for super admins
   - `SensitiveData` for super admins
4. It repairs persisted access if the user record exists but is missing required groups.

## Step 12: Umbraco completes backoffice sign-in

What happens:

1. Umbraco writes its own backoffice auth session.
2. The browser returns to the backoffice shell.
3. The browser starts calling backoffice APIs such as:
   - `https://localhost:5321/umbraco/management/api/v1/user/current`

## Step 13: Backoffice UI renders tabs based on Umbraco group membership

What happens:

1. Umbraco now uses its own local backoffice session, not the Identity SPA session.
2. Visible tabs and admin capabilities depend on Umbraco group membership.
3. If the user has:
   - `Administrators`
   - `SensitiveData`
   - and appropriate root access
4. the full admin backoffice tabs appear.

## Short Summary

1. browser opens `https://localhost:5321/umbraco`
2. Umbraco starts external login challenge
3. OIDC middleware redirects browser to `https://app.duchihao.com/identity/connect/authorize?...`
4. Identity sees user is not logged in and redirects to `https://app.duchihao.com/identity/login?...`
5. user signs in on the Identity login page
6. login page resumes `https://app.duchihao.com/identity/connect/authorize?...`
7. Identity issues authorization code and redirects to `https://localhost:5321/signin-oidc?...`
8. Umbraco validates token and claims
9. Umbraco provisions user and groups
10. Umbraco signs user into backoffice
11. backoffice UI renders based on Umbraco groups

## Best Files To Debug

If the problem is:

- redirect URI wrong
  - `OpenSaurBackOfficeAuthenticationExtensions.cs`
  - Identity OIDC client registration in Identity DB/admin UI

- browser does not leave Umbraco for Identity
  - `OpenSaurBackOfficeAuthenticationExtensions.cs`
  - Umbraco external login provider configuration

- browser reaches Identity but never returns to Umbraco
  - `OidcEndpoints.cs`
  - `IssuerAuthenticationFlow.cs`
  - Identity OIDC client redirect URI registration

- user authenticates but sees limited tabs
  - `OpenSaurBackOfficeUserProvisioningService.cs`
  - `OpenSaurIdentitySession.cs`

- user is rejected after successful token validation
  - `OpenSaurIdentitySession.TryCreate(...)`
  - claim copying in `OnTokenValidated`
